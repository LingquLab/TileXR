# Checker Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build `checker`, a no-NPU validation tool for TileXR collective operators that can run multi-rank functional checks, emit timing/event evidence, and produce fast, friendly problem localization without requiring production collective/EP operator source edits.

**Architecture:** Add an opt-in `tools/checker` subsystem with a small C++14 core library, deterministic rank world/memory model, collective oracles, evented shim/runtime primitives, ordering diagnostics, report writers, and a CLI. The MVP executes semantic multi-rank collective models for AllGather and AllReduce SUM on INT32, while preserving a compile/header-stub path for later source-preserving kernel harnesses.

**Tech Stack:** C++14, CMake/CTest, existing TileXR headers (`src/include`), no external test framework, no Ascend runtime dependency, JSON/text reports written by the checker core.

## Global Constraints

- Keep production collective and EP source unchanged. All checker-specific code must live under `tools/checker`, `tests/checker`, and root build wiring.
- Checker build is opt-in with `-DTILEXR_BUILD_CHECKER=ON`; default repo build behavior must not change.
- No CANN/Ascend runtime requirement for checker tests. The implementation may include CANN-compatible shim headers, but tests must pass on a host without NPU devices.
- Use C++14 only.
- Prefer deterministic test data and deterministic schedules (`serial`, `round_robin`) so failures are reproducible.
- Every failure report must include a top finding, affected rank/core when known, buffer/offset when known, and a concrete next action.
- Do not include EP behavior in MVP execution. Reserve extension points and add source-preservation tests so EP support can be added without changing the checker architecture.
- CANN CPU-domain twin debugging is optional reference material only. The MVP must not depend on it.
- Execute each implementation task with a test-first loop: add the listed test target, run it to observe failure or missing-target failure, implement the minimum code for that task, rerun the task verification command, then commit only that task's files.

std::string ReadFile(const std::string &path) {

    std::ostringstream out;

    return out.str();
}

void ExpectContains(const std::string &text, const std::string &needle,
                    const std::string &path) {
    if (text.find(needle) == std::string::npos) {
        std::cerr << path << " missing " << needle << "\n";
        ++g_failures;
    }
}
}  // namespace

    return g_failures == 0 ? 0 : 1;
